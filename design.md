# AI-Native Workshop Demo Design

## Purpose
This document translates the requirements in `spec.md` and `full_workshop_instructions.md` into a concrete design for a small, local, workshop-friendly AI-native application. The product is an AI Lab Assistant that accepts a technical assignment, runs a deterministic multi-step workflow, and presents the outputs in both a Streamlit UI and a CLI fallback.

## Design Goals
- Keep the codebase small and easy to teach in a 1-hour workshop.
- Run locally with open tools and local Ollama models only.
- Make the differences between prompt, workflow, and agent visible.
- Prefer deterministic orchestration over autonomous agent behavior.
- Support a reliable live demo with clear failure handling.

## Non-Goals
- Authentication or user accounts
- Persistent storage or databases
- Cloud deployment
- Advanced observability
- Complex agent memory or autonomous tool use

## Product Summary
The application accepts a short technical assignment such as CSV analysis and plotting. It sends that assignment through three role-based modules:

`User Input -> Planner -> Builder -> Reviewer -> Final Result`

Each role produces a distinct artifact:
- Planner: requirements, assumptions, implementation plan, edge cases
- Builder: starter Python code
- Reviewer: coverage analysis, gaps, risks, improvement suggestions

The coordinator assembles the final result into a structured object for display in the UI and terminal.

## Core Concepts Demonstrated
### Prompt
A single model interaction, such as asking the Planner to extract requirements.

### Workflow
A predefined sequence of model-backed steps with explicit data flow between them.

### Agent
A software module with one narrow responsibility, fixed inputs, and structured outputs.

## Users and Context
Primary users are students with basic Python knowledge and instructors running a live workshop. The system should optimize for readability, straightforward setup, and resilience when local models are slow or unavailable.

## Functional Design
### 1. Planner Agent
Responsibilities:
- Read the raw assignment
- Extract explicit requirements
- Identify assumptions
- Produce a concise implementation plan
- Note likely edge cases

Expected output structure:
- Requirements
- Assumptions
- Implementation Steps
- Edge Cases

Implementation note:
The Planner should return structured markdown-like text rather than free-form prose to simplify display and keep outputs predictable.

### 2. Builder Agent
Responsibilities:
- Use the original assignment and Planner output
- Generate readable starter Python code
- Include lightweight comments
- Avoid unnecessary abstraction

Expected output structure:
- Primarily code
- At most a short header or leading note if needed

Implementation note:
The Builder should be prompted to prefer a single-file example unless the assignment strongly suggests otherwise.

### 3. Reviewer Agent
Responsibilities:
- Compare the generated code against the assignment and plan
- Identify missing requirements
- Surface risks and edge cases
- Suggest practical improvements

Expected output structure:
- Coverage Check
- Missing Parts
- Risks / Edge Cases
- Improvement Suggestions

### 4. Coordinator
Responsibilities:
- Validate that assignment input is not empty
- Call Planner first
- Pass Planner output into Builder
- Pass assignment, Planner output, and Builder output into Reviewer
- Derive final next steps from the review content or simple formatting logic
- Return one final structured result

The coordinator should be deterministic, linear, and readable enough to explain live during the workshop.

### 5. LLM Client
Responsibilities:
- Read configuration from environment variables
- Call the Ollama HTTP API at `http://localhost:11434` by default
- Use `llama3` by default
- Provide timeout handling
- Surface clear errors when Ollama is down or the model is missing

Preferred interface:

```python
ask_model(system_prompt: str, user_prompt: str) -> str
```

The implementation should use a single reusable helper rather than separate request logic in each agent.

### 6. Result Assembly
The workflow should return a dictionary or dataclass containing:

```python
{
    "assignment": str,
    "requirements": str,
    "plan": str,
    "starter_code": str,
    "review": str,
    "next_steps": str,
}
```

`next_steps` can be generated by lightly transforming review suggestions instead of calling the model again.

## Architecture
### Execution Flow
1. User enters or loads an assignment
2. UI or CLI validates input
3. Coordinator runs Planner
4. Coordinator runs Builder with Planner output
5. Coordinator runs Reviewer with assignment, plan, and code
6. Coordinator assembles final result
7. UI or CLI renders the sections in order

### Module Responsibilities
- `app.py`: Streamlit interface for workshop demos
- `main.py`: CLI fallback for terminal-only runs
- `agents/*.py`: role-specific wrappers around prompts and model calls
- `workflow/coordinator.py`: deterministic orchestration
- `utils/config.py`: environment loading and defaults
- `utils/llm.py`: Ollama HTTP client
- `utils/formatting.py`: result shaping and display helpers
- `prompts/*.txt`: editable role instructions for teaching prompt design

## Interface Design
### Streamlit UI
The main screen should contain:
- Workshop title
- Short description of prompt vs workflow vs agents
- Text area for assignment input
- Run button

After execution, the page should show:
1. Assignment
2. Requirements
3. Plan
4. Starter Code
5. Review
6. Final Next Steps

Design principles:
- Keep the layout single-column and readable
- Use code blocks for Builder output
- Display clear warnings for empty input and connection issues
- Favor clarity over decorative UI

### CLI
The CLI should:
- Accept interactive input or `--input sample_data/assignment.txt`
- Run the same coordinator logic as the UI
- Print structured sections in the terminal

This provides a fallback if Streamlit setup is too slow or problematic during the workshop.

## Prompt Design
Each role should have a dedicated prompt file to make the system teachable and modifiable.

Guidelines:
- Keep prompts short and explicit
- Ask for structured outputs
- Avoid chain-of-thought requests
- Constrain the Builder toward simple, commented code
- Constrain the Reviewer toward concrete gaps and improvements

## Configuration Design
Environment variables:
- `MODEL_PROVIDER=ollama`
- `MODEL_NAME=llama3`
- `OLLAMA_BASE_URL=http://localhost:11434`

The code should use sensible defaults so `.env` is helpful but not strictly required.

## Error Handling
The system should handle these cases cleanly:
- Empty assignment input
- Ollama service not running
- Model not pulled locally
- HTTP timeout or connection failure
- Unexpected empty model response

Error messages should explain the likely cause and the next action, such as:
- start Ollama
- run `ollama pull llama3`
- verify the configured model name

## Non-Functional Design
- Python 3.10+
- Minimal dependencies: `streamlit`, `requests`, `python-dotenv`
- Optional dependencies only if they improve teaching value
- Small functions with type hints and concise docstrings
- Avoid heavy frameworks such as LangChain or LangGraph

## Sample Demo Scenario
Input:
> Build a Python program that reads sensor measurements from a CSV file, computes the average and standard deviation, and generates a simple plot.

Expected outputs:
- concise requirements summary
- step-by-step implementation plan
- readable starter Python code
- review feedback on coverage and risks
- final next-step suggestions

## Workshop Delivery Fit
This design supports the workshop goals by making:
- prompts visible through prompt files
- workflow visible through the coordinator
- agents visible through separate role modules
- local execution visible through Ollama-based model access

It also supports live teaching by providing both a UI path and a CLI fallback.

## Acceptance Criteria
The design is successful if the resulting project:
- runs locally without paid services
- connects to Ollama using configurable defaults
- produces Planner, Builder, and Reviewer outputs for a sample assignment
- shows all outputs clearly in Streamlit
- also runs from the CLI
- remains small enough for students to understand in one session

## Recommended Build Order
1. Create project structure and prompt files
2. Implement environment and Ollama client utilities
3. Implement Planner, Builder, and Reviewer modules
4. Implement the coordinator and result formatting
5. Build the CLI entry point
6. Build the Streamlit UI
7. Test with the sample assignment and failure scenarios

## Future Extensions
After the workshop, the project can be extended with:
- file upload support
- PDF task input
- retrieval over notes or documentation
- evaluation scoring
- branching workflows
- FastAPI exposure
- heavier orchestration frameworks if complexity later justifies them
